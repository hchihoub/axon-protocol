The Problem with MCP + Chrome

The Model Context Protocol (MCP) was designed for simple tool calls — read a file, run a query, return a string. It works well there. But browser automation is different. A single screenshot produces 50–200KB of base64 data. A page's accessibility tree can be thousands of nodes. Full page text extraction returns entire articles.

MCP dumps all of this directly into the model's context window. There's no compression, no summarization, no external storage. What happens when you run a realistic browser session?

The death spiral

3 screenshots alone consume ~189K tokens — nearly the entire 200K context window. Every subsequent tool result (pricing data, DOM trees, reviews) is silently truncated. The model loses the very data it needs to answer the user's question, then must re-fetch it, generating more screenshots that overflow context again.

How AXON Solves This

1. Three-Tier Lazy Discovery

Instead of dumping full tool schemas into context (MCP), AXON uses a 3-tier system: 

// Tier 1: Always in context (~20 tokens/tool)
{ id: "screenshot", summary: "Take a screenshot of the page", category: "capture", tags: ["screenshot","capture"] }

// Tier 2: Fetched on-demand when the model calls the tool (cached by content hash)
{ id: "screenshot", description: "Capture a PNG screenshot...", input: { type: "object", properties: { ... } } }

// Tier 3: Extended docs — only if explicitly requested
{ id: "screenshot", examples: [...], caveats: [...], related_tools: [...] }

2. Out-of-Context Result Store (OCRS)

The key innovation. Tool results are stored externally in a content-addressed store. The model gets a compact summary with a reference handle: 

// MCP injects into context: 61,304 tokens (raw base64)
{ "type": "image", "data": "iVBORw0KGgoAAAANSUhEUgAA...<240KB of base64>..." }

// AXON injects into context: 17 tokens (summary + ref)
[screenshot] Screenshot 1280x800 (180KB) of "ASUS ROG Strix G16" [ref:ax_r_amzn1]

// If the model needs the actual image, it requests:
detail(ref: "ax_r_amzn1")  // → Returns full data from OCRS

3. Capability-Based Security

// Every write operation requires an unforgeable capability token
{
  type: "resource:write",
  scope: "*.amazon.com",           // Only Amazon pages
  constraints: { max_calls: 100 }, // Rate limited
  expires_at: 1710432000           // TTL-based expiry
}

// Domain scoping blocks unauthorized navigation
navigate("https://evil.com")  // → Error: Domain not in allowed list

Real-World Case Study: The Schema Discovery Bug

During actual testing of MCP browser automation, we hit a bug that perfectly illustrates why AXON's architecture matters. Here's what happened:

The MCP Problem: Guessing Parameters

The agent needed to scroll a page. MCP had dumped all 18 tool schemas upfront (~4,900 tokens), but in a long session, the model's attention had drifted past them. It guessed the parameters based on the tool name alone: 

// What the agent sent (WRONG)
tools/call: {
  name: "scroll",
  arguments: { selector: "body" }  // ← Not a valid parameter!
}

// Error: "Tab 3 not found" (misleading error message)
// Real issue: wrong parameter schema entirely

The error was confusing because selector isn't even a parameter of the scroll tool. The actual schema requires tabId, direction, and amount. The agent had no way to know this without scrolling back through 4,900 tokens of upfront schema dumps.

vs
The AXON Solution: Schema on Demand

With AXON's 3-Tier Lazy Discovery, the agent always gets the right schema exactly when it needs it. Here's the same workflow: 

// Tier 1: Agent sees compact manifest in context (~20 tokens)
{ id: "scroll", summary: "Scroll page in a direction" }

// Tier 2: Agent requests schema BEFORE calling the tool
schema("scroll") → {
  input: {
    tabId:     { type: "number", required: true },
    direction: { type: "string", enum: ["up","down","left","right"] },
    amount:    { type: "number", description: "pixels to scroll" }
  }
}

// Agent calls with CORRECT parameters — first try
call("scroll", { tabId: 3, direction: "down", amount: 800 })
// ✓ Success

Why This Matters

🎯

First-Call Accuracy

AXON agents fetch the exact schema before each call. No guessing, no stale schemas buried in context, no misleading errors. The tool call works on the first attempt.

🧠

Attention Efficiency

MCP dumps 4,900 tokens of schemas upfront, and the model's attention fades over long sessions. AXON keeps only ~400 tokens of manifests in context — schemas are always fresh.

🔄

Zero Wasted Calls

The MCP bug required a retry after a confusing error. Each failed call wastes tokens, time, and money. AXON's on-demand schemas eliminate this entire class of errors.

Bottom line: MCP's "dump everything upfront" approach trades 4,900 tokens of context for schemas the model can't reliably attend to. AXON's lazy discovery uses 400 tokens of manifests + on-demand schemas, guaranteeing the agent always has the right parameters — with 92% less context.

Conclusion

MCP was designed for a world where tool results are small text strings. Browser automation doesn't live in that world. Screenshots, DOM trees, and page extractions produce results that are orders of magnitude larger than what MCP's architecture can handle.

The issue isn't just efficiency — it's correctness. When context overflows, the model loses the data it collected. It can't answer the user's question. It must re-fetch data, which generates more overflow. It's an architectural dead end.

AXON's OCRS solves this at the protocol level. Results live outside context. The model gets compact summaries that capture the essential information. When it needs detail, it does a surgical drill-down. The context window stays clean. The research quality stays at 100%.
